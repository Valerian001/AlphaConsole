# ADR-2026-04-30: Multi-Agent Runtime Initialization & Sandbox Lifecycle

## Status
Approved

## Context
While the Vast.ai instance handles the hardware level, the "several agents" (Planner, Reviewer, Developer, etc.) need a standardized way to be initialized, isolated, and managed within that instance. We need to define how these processes are spawned and how they receive their specific context and toolsets.

## Decision
1.  **Go Supervision Layer:** The **Go Execution Runtime** acts as the primary supervisor. It is the only process that communicates with NATS.
2.  **The "Agent Shell" Pattern:** Agents are not long-running services. They are spawned as **isolated Python processes** (Shells) on-demand by the Go Runtime.
3.  **Initialization Sequence:**
    *   **Trigger:** Go Runtime receives a `task.assigned` message from NATS.
    *   **Sandbox Creation:** Go creates an ephemeral execution directory and mounts the project volume.
    *   **Context Injection:** Go generates a "Runtime Manifest" (JSON) containing the agent's role, system prompt, and a short-lived JWT.
    *   **Process Spawn:** Go executes the Python Agent Shell, passing the Manifest path as an argument.
4.  **Agent Warm-Pool (Latency Optimization):**
    *   To avoid the ~1s Python startup lag, the Go Runtime maintains a "Warm Pool" of pre-initialized Python shells in an `IDLE` state.
    *   When a task arrives, Go "hydrates" an idle shell with the task-specific manifest.
    *   Developer agents (capable of running code) are spawned inside a **sidecar Docker container** via the Docker Engine API to ensure they cannot compromise the main instance.
    *   Planner/Reviewer agents (reasoning only) run as standard processes under `cgroups` limits.

## 6. Concurrency & Fleet Scaling Policy
*   **Task-to-Agent Ratio:** The system follows a **1:1 Parallel Execution** model. For $N$ independent tasks dispatched by the Reviewer, the Go Runtime will hydrate $N$ separate Developer Agent shells.
*   **Parallelism Limit:** The number of concurrent agents is dynamically governed by the `hardware_specs` (VRAM and RAM headroom).
*   **Sequential Fallback:** If a task has direct dependencies (Parent-Child relationship in the DAG), the child task remains in a `PENDING` state until the parent agent reports `SUCCESS`. 
*   **Inference Sharing:** Multiple concurrent agents share the local **Ollama** inference stream, utilizing NATS-based request queuing to prevent GPU OOM errors.

## Consequences
*   **Pros:** High isolation between agents. Predictable resource usage. Fast scaling within a single instance.
*   **Cons:** Complexity in managing the "Warm Pool" and the Go-Python bridge.
