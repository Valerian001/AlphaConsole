# ADR-2026-04-27: Implementation Strategy for Agile Features

This record outlines the technical decisions and implementation patterns for the core AlphaConsole features defined in the design phase.

---

## 1. Agile Planning Loop (Planner ↔ Reviewer ↔ User)

### 1.1 State Management (PostgreSQL & NATS)
*   **Decision:** We will use a "State Machine" pattern stored in the `objectives` table in Supabase.
*   **Mechanism:** 
    *   An objective will have statuses: `DRAFT`, `INTERNAL_REVIEW`, `AWAITING_HUMAN`, `RECYCLING`, `DISPATCHED`.
    *   Transitions between Planner and Reviewer are triggered via NATS subjects (e.g., `agent.planner.output`, `agent.reviewer.feedback`).
    *   The "Recycle" loop increments an `iteration_count` in the database to track the agile flow.

### 1.2 Communication Protocol
*   **Decision:** Use JSON-Schema for all agent-to-agent messages to ensure structured data transfer.
*   **Logic:** The Reviewer agent will utilize a specialized "Critique System Prompt" to analyze the Planner's JSON output.

---

## 2. Reviewer as the Dispatcher (Fleet Orchestration)

### 2.1 Fleet Awareness Implementation
*   **Decision:** The Reviewer agent will have a "Capability Discovery" tool.
*   **Mechanism:** Upon initialization, the Reviewer queries the `agents` table in PostgreSQL to pull the current fleet's capability matrix.
*   **Dispatching:** The Reviewer publishes task assignments to NATS subjects specific to the executing agent types (e.g., `task.developer.assigned`).

---

## 3. Task Evolution Engine (Real-time DAG)

### 3.1 Data Structure
*   **Decision:** Tasks are stored in a self-referencing `tasks` table with a `parent_task_id`.
*   **Real-time Sync:** The FastAPI Control Plane will use PostgreSQL Listen/Notify or a NATS-to-WebSocket bridge to push updates to the AlphaConsole frontend.
*   **VRAM Telemetry:** The Control Plane will aggregate worker telemetry to provide a tri-metric view: `Total`, `Used`, and `Remaining` VRAM, sourced from `nvidia-smi --query-gpu=memory.total,memory.used,memory.free --format=csv`.

---

## 4. Ephemeral-to-Persistent Data Offloading

### 4.1 Sync Mechanism
*   **Decision:** Use a "Sidecar Sync" strategy.
*   **Mechanism:** 
    *   **Postgres:** Ephemeral worker DB is dumped via `pg_dump` and streamed directly to Supabase before shutdown.
    *   **MinIO:** Local artifacts are mirrored to Contabo using the `mc mirror` CLI tool in a dedicated teardown container.
    *   **Qdrant:** Collection snapshots are generated and uploaded to the Contabo-hosted MinIO bucket.

---

## 5. Security & Identity

### 5.1 JWT Implementation
*   **Decision:** Each agent is issued a short-lived JWT by the Control Plane at the start of an objective.
*   **Scope:** Claims in the JWT limit the agent's access to specific NATS subjects and MinIO buckets (e.g., `scope: ["task_id_123:*"]`).

---

## 6. Development Priorities (Build Order)

1.  **Core NATS/Postgres State Machine:** The foundation for the agile loop.
2.  **Reviewer Agent Intelligence:** Implementing the "Critique & Dispatch" logic.
3.  **Vast.ai Orchestrator CLI:** Enabling the spin-up/spin-down of workers.
4.  **AlphaConsole Frontend:** Building the interactive dashboard components.
