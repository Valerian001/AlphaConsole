# ADR-2026-04-28: Manual Instance Provisioning & Vast.ai Hardware Selection

## Status
Proposed

## Context
Vast.ai instances are ephemeral and short-lived. To maximize cost efficiency and ensure the correct hardware is used, the system needs a way to spawn new instances on-demand. The user has requested a manual selection step where they can choose a server based on price after the system filters for the required hardware specifications (NVIDIA RTX 4090, 24GB VRAM, etc.).

## Decision
1.  **Provisioning Trigger:** If no active worker instance is detected by the Control Plane, the Dashboard will transition to a "Restricted" state, prompting the user to provision a new server.
2.  **Hardware Filtering:** The Control Plane will query the Vast.ai API/CLI with the following filters:
    *   `gpu_name == RTX_4090`
    *   `gpu_ram >= 24`
    *   `inet_up >= 500` (High bandwidth)
    *   `cpu_cores >= 16`
3.  **Price-Based Selection UI:** The system will present a list of available servers matching the criteria, sorted by price ($/hr). The user must select one to proceed.
4.  **Automated Deployment:** Once a server is selected, the Control Plane will:
    *   Execute `vastai create instance`.
    *   Inject the `onstart` bootstrap script (ADR-2026-04-26).
    *   Track deployment progress (Booting -> Installing -> Ready).

## Overrides
*   Overrides the "Automated Provisioning" mentioned in `Instance_Lifecycle_Specs.md` to include the mandatory **Human-in-the-loop Pricing Gate**.

## Consequences
*   **Pros:** Prevents unexpected costs by allowing users to choose the cheapest available spot instance. Ensures high-performance hardware is always used.
*   **Cons:** Introduces a small manual delay before a project can start.
